### Root Cause Analysis

#### Critical Bug #1: Sentinel permanently paused (FOUND THIS CYCLE)
`FullSentinel.system_healthy()` checks `self._is_paused` but **never calls `_check_pause_expired()`**. When anomaly detection triggers a timed pause (e.g., 1 hour), the `validate_plan()` path does check expiry, but `system_healthy()` does not. Since the Phase3 orchestrator calls `system_healthy()` first and returns early when it's False, `validate_plan()` never runs, and the pause never expires.

**Impact:** The main orchestrator has been stuck in "System not healthy — skipping cycle" for an unknown number of cycles. No stop-losses, no new trades, no exit management — positions are completely unprotected. This explains why ETH (-9.5%), SOL (-2.0%), AVAX (-12.7%), DOT (-12.5%) are all far past the 2% stop-loss threshold without being exited.

**Evidence:** Cloud Run logs show `"System not healthy - skipping cycle"` at every main-cycle trigger (e.g., 08:05:53 UTC Cycle #59). Meme trader cycles continue running (separate scheduler) but its sell orders also fail ("0 successful, 1 failed").

#### Critical Bug #2: Stop-losses gated behind health check
Even without Bug #1, the orchestrator's architecture has a design flaw: `check_stop_losses()` runs AFTER the `system_healthy()` gate. This means any sentinel pause (anomaly, circuit breaker, manual) also disables all exit protection. Stop-losses are a safety mechanism and should never be gated.
