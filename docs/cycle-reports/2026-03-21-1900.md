## Root Cause Analysis

### Critical Bug: Meme trades never recorded to database

**Root cause:** `MemeOrchestrator._execute_signal()` builds a `Trade` object and manages in-memory position state, but never calls `self.memory.record_trade()`. The main orchestrator (Phase3) does call this after every trade.

**Impact chain:**
1. Direct-sell trades (including stop-loss exits) bypass the executor
2. No `record_trade()` call → DB has BUY records but no SELL records
3. On Cloud Run restart, `_reconstruct_positions_from_db()` queries DB
4. Sees net_buy > 0 for sold positions → recreates "zombie" positions
5. Zombie positions accumulate with each restart
6. Current state: 13 meme zombie positions at -75% to -99.99% loss
7. Sentinel calculates -$5,182 daily meme PnL from these zombies
8. Circuit breaker trips repeatedly on phantom losses

### Secondary: Anthropic API credits exhausted

- Falling back to Codex — functional but lower quality signals
- Not a code issue — needs credit top-up or model switch

## Implemented Fix

### Fix 1: Record meme trades to DB (commit `cb99bf8`)
- Added `self.memory.record_trade(trade, intel)` call immediately after trade execution in `_execute_signal()`, before position cleanup
- Applies to ALL trade actions (BUY and SELL) from both direct-sell and executor paths
- Graceful degradation: if memory store unavailable, logs warning but doesn't block trade
- Creates minimal `MarketIntel` with regime=VOLATILE for DB schema compatibility

**Risk:** Low — additive DB write, no changes to execution logic
**Reversibility:** Full — removing the write doesn't affect trading behavior
**Files changed:** 1 (`agents/memetrader/orchestrator.py`)
**Tests:** 151/151 pass (excl. 1 pre-existing Binance test)
