## Root Cause #1: Consecutive Loss Circuit Breaker Deadlock (CRITICAL)

**Bug:** `CircuitBreakers.record_trade()` trips the `consecutive_loss` breaker after 3+ consecutive losing trades with `reset_on_win=True` but **no time-based `reset_at`**. In `check_all()`, the auto-reset check `if breaker.reset_at and now > breaker.reset_at` evaluates to False (reset_at is None). The breaker can ONLY clear via `record_trade(pnl > 0)` — but the breaker blocks all trading, so no winning trade can ever occur.

**Impact:** Permanent trading deadlock after 3 consecutive losses. Main bot has been stuck in this state, unable to enter or exit positions through normal flow.

**Fix:** Added `reset_hours=4` as time-based fallback when tripping consecutive_loss breaker. Breaker now auto-resets after 4 hours OR on a winning trade, whichever comes first.

**Risk:** Low — this only adds an escape valve. The breaker still reduces position sizing and blocks trading for 4 hours after consecutive losses, which is sufficient protection.

## Root Cause #2: Meme Sell Phantom Trades

**Bug:** `MemeOrchestrator._execute_signal()` calls `exchange.market_sell()` directly for tracked positions. When the simulation exchange has no balance, it returns `{"error": "No position"}`. The orchestrator doesn't check the `error` key — it falls through to `result.get("price") or last_price`, creating a phantom trade with last_price as fill. Position tracking updates as if the sell succeeded.

**Impact:** Meme positions appear to close but don't actually sell. PnL is phantom. Positions accumulate indefinitely in the sim exchange.

**Fix:** Added error-key check on `market_sell()` result. If `result.get("error")` is truthy, log warning and return None (no trade recorded).

**Risk:** Low — only affects sells that were already failing silently.
